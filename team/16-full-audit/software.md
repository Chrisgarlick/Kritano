# Software Audit

**Overall Assessment:** STRONG
**Score:** 8/10

## What's Working Well

1. **Parameterized queries throughout** -- Every SQL query across `user.service.ts`, `token.service.ts`, `audit.service.ts`, `job-queue.service.ts`, and route handlers uses `$1, $2, ...` placeholders. No string interpolation into SQL was found anywhere. This eliminates SQL injection risk.

2. **Robust authentication architecture** -- Refresh token rotation with reuse detection (`token.service.ts:161-176`), token family revocation on suspected theft, absolute expiry caps that prevent infinite session extension, and Argon2id password hashing with OWASP-recommended parameters. This is production-grade auth.

3. **Defense-in-depth security layering** -- Helmet with strict CSP (`security.middleware.ts`), CSRF double-submit cookie pattern with timing-safe comparison (`csrf.middleware.ts`), Redis-backed rate limiting with in-memory fallback (`rateLimit.middleware.ts`), account lockout after 5 failed attempts, SSRF protection on URL inputs (`ip.utils.ts`), and Sentry cookie stripping. Each layer independently mitigates real attack vectors.

4. **Anti-enumeration patterns** -- Password reset and email verification endpoints return identical success messages regardless of whether the email exists (`auth/index.ts:658, 621`). This prevents user enumeration attacks.

5. **Webhook security** -- Raw body parsing registered before JSON middleware (`index.ts:78-79`) to preserve signature verification integrity for Stripe and Resend webhooks. CSRF is correctly skipped for webhook paths.

## Issues Found

### 1. Change Password Uses Wrong User ID Property
**Severity:** CRITICAL
**Location:** `server/src/routes/auth/index.ts:762`
**Finding:** The change-password handler reads `(req as any).userId` instead of `req.user!.id`. The `authenticate` middleware sets `req.user.id`, not `req.userId`. This means `userId` is always `undefined`, and `findById(undefined)` will fail or return null, making the change-password feature completely non-functional.
**Impact:** Users cannot change their password through the authenticated endpoint. If `findById` silently returns null, the handler returns a 400 error with no useful explanation. More critically, if any other middleware or code path does set `req.userId` to a different value, this could allow password changes on the wrong account.
**Recommendation:** Change line 762 from `const userId = (req as any).userId;` to `const userId = req.user!.id;`

### 2. Change Password Missing Strength Validation
**Severity:** HIGH
**Location:** `server/src/routes/auth/index.ts:755-788`
**Finding:** The change-password handler validates the current password and updates to the new password, but it does not call `passwordService.validateStrength()` or `passwordService.isCommonPassword()` on the new password. Both the register and reset-password flows enforce these checks.
**Impact:** Authenticated users can change their password to weak values like "a" or common passwords like "password123", bypassing all password policy. This undermines the security posture established by the other flows.
**Recommendation:** Add `validateStrength(newPassword)` and `isCommonPassword(newPassword)` checks before calling `updatePassword()`, matching the pattern used in the reset-password handler.

### 3. timingSafeEqual Leaks Length Information
**Severity:** MEDIUM
**Location:** `server/src/utils/crypto.utils.ts:28-29`
**Finding:** The `timingSafeEqual` function returns `false` immediately when string lengths differ. This early return reveals whether the CSRF token length matches via timing side-channel.
**Impact:** For CSRF double-submit cookies, both tokens should always be the same length (generated by the same function), so this is low practical risk. However, if this utility is reused for other comparisons in the future, it could leak information. The standard mitigation is to hash both inputs before comparing.
**Recommendation:** Either (a) document that this function must only be used with equal-length inputs, or (b) hash both inputs with SHA-256 before comparing, which normalizes length.

### 4. CORS Origin Is a Single Value, Not a List
**Severity:** LOW
**Location:** `server/src/index.ts:70`
**Finding:** `CORS_ORIGIN` is treated as a single origin string. If the app needs to serve multiple origins (e.g., `www.pagepulser.com` and `pagepulser.com`, or a staging domain), there is no mechanism for that.
**Impact:** Currently fine for a single-domain deployment, but will require code changes if multiple origins are needed (e.g., staging environments).
**Recommendation:** Consider parsing `CORS_ORIGIN` as a comma-separated list and using a function-based origin validator.

### 5. Duplicate Pool Instantiation
**Severity:** LOW
**Location:** `server/src/index.ts:48-51` and `server/src/db/index.ts:6-11`
**Finding:** Two separate `Pool` instances are created -- one in `db/index.ts` (imported throughout services) and one in `index.ts` (passed to `initializeRoutes`). Both connect to the same database but maintain independent connection pools.
**Impact:** The application uses up to 40 PostgreSQL connections instead of 20 (two pools with `max: 20`). On a single DigitalOcean droplet, this wastes memory and could hit the PostgreSQL `max_connections` limit under load.
**Recommendation:** Remove the second `Pool` in `index.ts` and pass the existing `pool` from `db/index.ts` to `initializeRoutes()`. Eliminate the `setPool()` pattern in favor of importing the shared pool directly.

### 6. Access Token Expiry Is 4 Hours
**Severity:** LOW
**Location:** `server/src/config/auth.config.ts:14`
**Finding:** Access tokens are valid for 4 hours (`14400` seconds). OWASP recommends 15-30 minutes for access tokens when refresh token rotation is implemented.
**Impact:** If an access token is stolen, the attacker has a 4-hour window to use it. Since access tokens are stateless JWTs, they cannot be revoked before expiry.
**Recommendation:** Reduce `accessTokenExpiry` to 900 (15 minutes) or 1800 (30 minutes). The refresh token rotation mechanism already supports transparent re-authentication.

## Opportunities

1. **Enable `noUncheckedIndexedAccess` in tsconfig** -- The TypeScript config has `strict: true` but lacks `noUncheckedIndexedAccess`. Enabling it would catch potential `undefined` access on array indexing and object lookups (e.g., `result.rows[0]?.field` patterns that currently lack guards). This would have caught the `(req as any)` cast pattern at compile time.

2. **Consolidate the `setPool()` pattern** -- `routes/index.ts` calls 18 separate `setPool()` functions to propagate the database connection. This is fragile and adds boilerplate. Migrating to a shared singleton import (`import { pool } from '../db/index.js'`) across all services would simplify initialization and eliminate the duplicate pool issue.

3. **Add structured logging** -- The codebase uses `console.log`/`console.error` throughout. Adopting a structured logger (pino or winston) would enable JSON log output, log levels, request correlation IDs, and easier log aggregation. The login timing debug logs (`index.ts:325`) should be behind a debug flag rather than always-on.

4. **Implement request timeout middleware** -- There is no per-request timeout. Long-running database queries or external API calls could hold connections indefinitely. Adding a middleware that aborts requests after 30 seconds would improve resilience.

5. **Add database connection health monitoring** -- The `testConnection()` function in `db/index.ts` runs once at startup but there is no ongoing health check. The pool `on('error')` event is not handled, meaning a dropped connection could cause silent failures. Adding pool error handlers and periodic health checks would improve reliability.

## Summary

PagePulser's security foundation is genuinely strong. The authentication system -- Argon2id hashing, refresh token rotation with reuse detection, CSRF protection, rate limiting, account lockout, anti-enumeration responses, and SSRF guards -- reflects thoughtful security engineering well above what most SaaS projects at this stage implement. The one critical issue is that the change-password endpoint is broken due to reading from a non-existent request property (`req.userId` instead of `req.user.id`), and it also skips password strength validation. The duplicate pool instantiation should be cleaned up to avoid wasting database connections. The codebase is well-organized with consistent patterns, parameterized queries everywhere, and proper error handling. Addressing the critical and high items would bring this to a 9/10.
